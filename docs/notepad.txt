### Device Simulator
**Status**: Core infrastructure COMPLETE. Frame format validated against ramses_tx.
**Location**: `features/device_simulator/IMPLEMENTATION_PLAN.md`

Core components:
- Message database (parses `ramses_rf/tests/fixtures/regression_packets_sorted.txt`)
- Virtual gateway (MQTT endpoint on `RAMSES/GATEWAY_SIM/18:001234`)
- Response engine (handles RQ → RP, including 2411 parameters)
- Periodic emitter (I frames at configured intervals)
- Scenario engine (device playback, flooding, timeout, conversation tests)

**Key Implementation Details** (see IMPLEMENTATION_PLAN.md "Frame Format Requirements"):
- I frames: `082  I --- src --:------ src code len payload` (extra space after verb)
- RP frames: `000 RP --- src dst --:------ code len payload` (no extra space)
- 2411 parameters: DB lookup by parameter ID, realistic payload values

### ha-sim Simulator Isolation Prerequisites
**For device_simulator feature to work with automatic isolation:**

1. **MQTT Integration in ha-sim**
   - Must be configured in `configuration.yaml` or UI:
   ```yaml
   mqtt:
     broker: 192.168.40.11 # Replace with your MQTT broker IP
     port: 1883
     username: "your_username" # Replace with your MQTT username
     password: "your_password" # Replace with your MQTT password
   ```
   - Required for MqttEndpoint to subscribe/publish

2. **ramses_cc Config (auto-enforced)**
   - Must use MQTT transport (not serial)
   - Device simulator automatically reconfigures to:
     - Topic: `RAMSES/GATEWAY_SIM/18:001234`
     - HGI: `18:001234` (fake, isolated from production)
   - note: the topic namespace is hardcoded in the simulator code.
   in _rf there is a strict check on prefix `RAMSES/GATEWAY` and also 3 parts divided by '/'
   but we can add the _SIM suffix to the topic to pass the check.

3. **MQTT Broker Permissions**
   - User must have subscribe/publish rights to `RAMSES/GATEWAY_SIM/#`
   - Same broker credentials as production

4. **Port Configuration**
   - ha-sim uses port 8124 (not 8123)
   - Set in `configuration.yaml`:
   ```yaml
   http:
     server_port: 8124
   ```

**Isolation Behavior:**
- Production devices use: `RAMSES/GATEWAY/18:xxxxxx`
- Simulator uses: `RAMSES/GATEWAY_SIM/18:001234`
- Zero crosstalk between namespaces

- **Migrate frontend message delivery to Event Entity (with backward compat)** — DONE:
  ramses_cc >= 0.55.6 has a proper HA Event Entity (`event.ramses_cc_regex_event`) that fires
  for every inbound packet when `message_events` regex is configured (e.g. `.*`). This is the
  modern HA-recommended way (see ramses_cc issue #552) — it fires as `state_changed`, which is
  in HA's websocket SUBSCRIBE_ALLOWLIST, so any user can subscribe (not just admin).
  Implementation (ramses-message-broker.js):
  - Modern path: subscribes to `state_changed`, filters for `event.ramses_cc_regex_event`,
    extracts data from `new_state.attributes.data` (same code/src/verb/payload structure).
  - Legacy path: also subscribes to `ramses_cc_message` bus events (for ramses_cc < 0.55.6
    or when message_events is not configured). May fail for non-admin — expected.
  - Dedup: 5s TTL Map keyed by `src|code|dtm` prevents processing the same message twice
    when both paths are active.
  - Backend (`ramses_message_stream.py`) unchanged — keeps firing bus events for backward compat
    and keeps `add_msg_handler` for backend automations/traffic collector.
  - Verified: non-admin websocket test receives 31DA/10D0/22F1/31D9 events via the modern path.

- **TIMING RACE**: `SIGNAL_UPDATE` in `event.py` (`async_process_msg`) fires synchronously
  from the protocol's extra handler list, BEFORE the Gateway's async `_msg_handler` (which
  does CQRS state ingestion via `create_task`) has a chance to run. Result: entities receive
  the signal, call `async_write_ha_state`, read `native_value` → `hvac_state`, but `hvac_state`
  is still stale because the ingestion task hasn't executed yet.
  - **Fix attempt**: defer signal send with `asyncio.sleep(0)` via `async_create_task` —
    signals ARE sent and entities ARE updated, but DB still shows stale values (sleep(0) may
    not yield enough). Needs deeper investigation — possibly need to await the actual
    ingestion task, or have the Gateway send the signal after ingestion completes.
  - **Root cause location**: `ramses_tx/protocol/base.py:_msg_received()` calls
    `self._msg_handler(msg)` first (async, scheduled via `create_task`), then iterates
    `self._msg_handlers` (sync callbacks including the event entity). The sync callback
    runs immediately, before the async task gets a timeslice.
  - **Separate from ramses_rf PR #732** (which fixes state ingestion itself). This is a
    ramses_cc issue that needs its own PR.

## PREP: Recipe 32 — Battery (1060) cache restore regression (issue 822 comment)

**Status**: DESIGN ONLY — do NOT edit ha_sim_test.py yet (another thread is working on it).
**Trigger**: https://github.com/ramses-rf/ramses_cc/issues/822#issuecomment-5011291025
            (peternash: "battery sensors not being cached (or at least only a small
            percentage) and going Unknown again after a period of time")

### Root cause (CONFIRMED — same pattern as the 0004/HGI fix already landed)

1. `Code._1060` (device_battery) is in `HIGH_VOLUME_STATUS_CODES`
   (ramses_rf `src/ramses_rf/const.py:165`).
2. 1060 is sent by battery-powered devices only once per day
   (`"lifespan": td(days=1)` in `protocol_schema.py:191`).
3. `_restore_cached_packets` (ramses_rf `lifecycle.py:285-315`) skips any
   cached packet older than 1h that is in `HIGH_VOLUME_STATUS_CODES`
   (`cutoff_dtm = now - 1h`; `if is_match: continue`).
4. ramses_cc keeps cached packets for 24h (`coordinator.py:279`
   `if dt_obj <= now - td(days=1): continue`), so a 2-24h-old 1060 survives
   in the ramses_cc cache but is then dropped by ramses_rf on restore.
5. Result: on restart, the only cached 1060 is almost always >1h old →
   skipped → message_store has no 1060 → `RamsesBatteryBinarySensor`
   state = Unknown. Only devices that happened to send a 1060 within the
   last hour before restart keep their battery reading ("small percentage").

### Fix (mirrors the 0004 fix already in const.py:159-163)
Remove `Code._1060` from `HIGH_VOLUME_STATUS_CODES` in
`/home/willem/dev/ramses_rf/src/ramses_rf/const.py`. 1060 is low-volume
(1/day) and carries semi-static state — exactly the same mis-classification
that caused the 0004 zone-name regression.

### Recipe 32 design (to be appended after Recipe 31 in ha_sim_test.py)

Goal: prove the bug, then prove the fix, by simulating the real-world
"cached 1060 is >1h old on restart" condition. The ha-sim TRV emits 1060
every 13.5s (TRV.yaml), so the bug does NOT manifest naturally — we must
age the cached 1060 timestamps in `.storage/ramses_cc` before restarting.

    # RECIPE 32: Battery (1060) cache restore — stale 1060 must survive
    # =====================================================================
    # Same regression class as issue 822 / 0004 zone names: 1060 is in
    # HIGH_VOLUME_STATUS_CODES but is sent only 1/day, so a cached 1060
    # older than 1h is skipped by _restore_cached_packets on restart and
    # the battery binary sensor goes Unknown.  We force the real-world
    # condition by ageing the 1060 timestamps in .storage/ramses_cc to
    # >1h ago, then restart ha-sim and check the battery entity.
    log_section("Recipe 32: Battery 1060 cache restore (stale >1h, issue 822)")

    # 1. Load mixed profile + activate CTL/TRV/DHW for heartbeats
    await load_profile_yaml(token, _mixed_yaml(), speed=0.01)
    wait(15, "for ramses_cc reload"); token = get_token(); wait(5, "init")
    for dev_id, name in [(CTL,"CTL"),(TRV,"TRV"),(DHW,"DHW")]:
        await ws_send(token, {"type":"ramses_extras/device_simulator/activate_profile_device","device_id":dev_id})
    wait(15, "for 1060 battery packets to populate message_store")

    # 2. Force a fresh 1060 I from TRV (battery 100%, low=0) — payload 006400
    #    schema: ^0[0-9A-F](FF|[0-9A-F]{2})0[01]$  (idx, level, low_flag)
    call_service(token, "ramses_extras", "device_simulator_inject_message",
        {"source_id": TRV, "code": "1060", "payload": "006400", "verb": "I"})
    wait(5, "for 1060 to process")
    call_service(token, "ramses_cc", "force_update")
    wait(5, "for entity state write")

    # 3. Verify the TRV battery binary sensor is ON (battery_ok) before restart
    entities = get_entities(token)
    bat = find_entity_for_device(entities, TRV, prefix="battery_")
    bat_state_before = bat.get("state") if bat else None
    check("TRV battery binary sensor has state before restart",
          bat is not None and bat_state_before in ("on","off"),
          f"state={bat_state_before!r} entity={bat.get('entity_id') if bat else None}")

    # 4. Age every cached 1060 packet to 2h ago in .storage/ramses_cc.
    #    Must stop ha-sim first (HA storage is not safe to write while up):
    #    docker stop ha-sim → rewrite .storage/ramses_cc → docker start ha-sim
    aged_ts = (dt_util.now() - td(hours=2)).isoformat(timespec="microseconds")
    storage = get_ramses_storage()
    packets = storage.get("client_state", {}).get("packets", {})
    n_1060 = sum(1 for p in packets.values() if isinstance(p, dict) and p.get("code")=="1060")
    check("Cached 1060 packets exist to age", n_1060 > 0, f"found {n_1060} 1060 in cache")
    new_packets = {}
    for ts, pkt in packets.items():
        if isinstance(pkt, dict) and pkt.get("code") == "1060":
            new_packets[aged_ts] = pkt   # collapse all 1060 to one aged ts
        else:
            new_packets[ts] = pkt
    storage["client_state"]["packets"] = new_packets
    # stop, write, start
    subprocess.run(["docker","stop","ha-sim"], capture_output=True)
    _write_ramses_storage(storage)  # helper to add: docker cp temp file → /config/.storage/ramses_cc
    subprocess.run(["docker","start","ha-sim"], capture_output=True)
    wait(20, "for ha-sim startup"); log_monitor.reset_baseline(); token = get_token()
    wait(10, "for ramses_cc to restore cached packets + create entities")

    # 5. WITHOUT FIX: battery entity = Unknown (1060 >1h skipped)
    #    WITH FIX (1060 removed from HIGH_VOLUME_STATUS_CODES): state retained
    entities = get_entities(token)
    bat_after = find_entity_for_device(entities, TRV, prefix="battery_")
    bat_state_after = bat_after.get("state") if bat_after else None
    check("TRV battery binary sensor retained state after restart (stale 1060)",
          bat_after is not None and bat_state_after in ("on","off"),
          f"state={bat_state_after!r} (Unknown/None = bug present)")
    # Expected: FAIL on ramses_rf master (bug), PASS after the const.py fix.
