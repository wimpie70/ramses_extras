### SLOW STARTUP: get_all_fan_params blocks HA startup for 200+ seconds

**Symptom:** HA takes 220s to initialize after restart with ramses_cc + FAN devices.

**Root cause:** `fan_handler.py:257` calls `coordinator.get_all_fan_params(call)`
unconditionally during setup for every FAN device. This sends 16 RQ 2411 commands
with 0.5s gaps. At the same time, the ramses_rf DiscoveryService is sending RQ
2E04, 2349, 22F8, etc. to all devices. All commands go through the same QoS
queue (single transport). When a discovery command times out (20s), it blocks
the queue — the 2411 commands sitting behind it can't be sent. With 8 devices
being probed and multiple timeout cycles, the queue gets backed up for 200+ seconds.

**Fix:** Remove the unconditional `get_all_fan_params` call at `fan_handler.py:257`.
The `on_fan_first_message` callback at line 201 already calls `get_all_fan_params`
after the FAN sends its first message — this is the correct trigger. The
unconditional call at line 257 is redundant and causes QoS queue contention
with the discovery service during startup.

**Branch:** Create new branch in ramses_cc for this fix (after PR 759 is merged).

- **TIMING RACE**: `SIGNAL_UPDATE` in `event.py` (`async_process_msg`) fires synchronously
  from the protocol's extra handler list, BEFORE the Gateway's async `_msg_handler` (which
  does CQRS state ingestion via `create_task`) has a chance to run. Result: entities receive
  the signal, call `async_write_ha_state`, read `native_value` → `hvac_state`, but `hvac_state`
  is still stale because the ingestion task hasn't executed yet.
  - **Fix attempt**: defer signal send with `asyncio.sleep(0)` via `async_create_task` —
    signals ARE sent and entities ARE updated, but DB still shows stale values (sleep(0) may
    not yield enough). Needs deeper investigation — possibly need to await the actual
    ingestion task, or have the Gateway send the signal after ingestion completes.
  - **Root cause location**: `ramses_tx/protocol/base.py:_msg_received()` calls
    `self._msg_handler(msg)` first (async, scheduled via `create_task`), then iterates
    `self._msg_handlers` (sync callbacks including the event entity). The sync callback
    runs immediately, before the async task gets a timeslice.
  - **Separate from ramses_rf PR #732** (which fixes state ingestion itself). This is a
    ramses_cc issue that needs its own PR.

## TODO: ramses_rf follow-up — discovery scan callback for real-time notifications

**Context:** ramses_rf's DiscoveryScan is purely polling-based — no callback/event
when a new device is first seen. ramses_cc currently polls every 30 min (plus an
immediate check 10s after startup) to detect new devices and send persistent
notifications. This means a user waits up to 30 min in steady state before being
notified of a new device.

**Goal:** Add a callback/listener mechanism to DiscoveryScan in ramses_rf so
ramses_cc can register an `on_new_device` callback and get notified in real time
when the scan engine classifies a new device.

**Where to add it (ramses_rf):**
- `src/ramses_rf/discovery_scan.py` — DiscoveryScan class
- Add a `callbacks: list[Callable[[DiscoveredDevice], None]]` param or
  an `add_listener(callback)` method
- Fire the callback when a device transitions from unknown → classified
  (i.e., when `likely_type` is first assigned)

**Where to use it (ramses_cc):**
- `custom_components/ramses_cc/discovery.py` — DiscoveryManager
- Register a listener at startup that calls `check_for_new_devices()` +
  sends the persistent notification immediately
- Keep the 30-min checkpoint for lost-device detection and state persistence

**Prerequisite:** The ramses_rf PR that introduced DiscoveryScan was just merged.
This callback would be a follow-up PR on ramses_rf.

**Priority:** Medium — the 10s immediate check + 30 min polling works, but
real-time notification is much better UX for the discovery review flow.
